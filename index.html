核心类 DispatcherServlet

客户端发送一个url请求被服务器接收后，都会被注册在web.xml中的DispatcherServlet接收，DispatcherServlet的本质是Servlet，其作用就是接受和转发web请求到内部框架处理单元（controller）。

上图可以看出DispatcherServlet的继承关系，Servlet的生命周期包含三部分：init、service、destroy。而DispatcherServlet的init方法定义在父类HttpServletBean中，且该方法为final。

DispatcherServlet的初始化

initServletBean()在HttpServletBean中并没有具体实现，需要HttpServletBean的子类来实现。这属于模板方法设计模式，即init作为模版方法。

init-param 元素读取

try-catch主要作用是：

  • 注册一个字符串到资源文件的编辑器，让Servlet下面的配置元素可以使用形如“classpath:”这种方式指定SpringMVC框架bean配置文件的来源。
  • 将web.xml中在DispatcherServlet这个Servlet下面的配置元素利用JavaBean的方式（即通过setter方法）读取到DispatcherServlet中来。

HttpServletBean这个类的设计中，运用了依赖注入思想完成了 init-param 配置元素的读取，这是一种setter注入。明白了HttpServletBean类的设计思想，我们也就知道可以如何从中获益。具体来说，我们继承HttpServletBean类（就像DispatcherServlet做的那样），在类中定义一个属性，为这个属性加上setter方法后，我们就可以在元素中为其定义值。在类被初始化后，值就会被注入进来，我们可以直接使用它，避免了样板式的getInitParameter()方法的使用，而且还免费享有Spring中资源编辑器的功能，可以在web.xml中，通过“classpath:”直接指定类路径下的资源文件。

容器上下文的建立

SpringMVC使用了Spring容器来容纳自己的配置元素，拥有自己的bean容器上下文。在SpringMVC初始化的过程中，非常关键的一步就是要建立起这个容器上下文，而这个建立上下文的过程就是依赖直接继承自HttpServletBean的FrameworkServlet类。HttpServletBean中的initServletBean()方法没有具体实现，FrameworkServlet重写了该方法，并且是final的。

核心代码只有try-catch中的两行，而initFrameworkServlet()没有具体实现，所以initServletBean()主要工作就是

this.webApplicationContext = initWebApplicationContext();

说明FrameworkServlet这个类的设计目的就是抽离出来建立WebApplicationContext上下文这个过程。initWebApplicationContext()方法，封装了建立Spring容器上下文的整个过程，方法内的逻辑如下：

  • 获取由ContextLoaderListener初始化并注册在ServletContext中的根上下文，记为rootContext
  • 如果webApplicationContext已经不为空，表示这个Servlet类是通过编程式注册到容器中的（Servlet 3.0+中的ServletContext.addServlet() ），上下文也由编程式传入。若这个传入的上下文还没被初始化，将rootContext上下文设置为它的父上下文，然后将其初始化，否则直接使用。
  • 通过wac变量的引用是否为null，判断第2步中是否已经完成上下文的设置（即上下文是否已经用编程式方式传入），如果wac==null成立，说明该Servlet不是由编程式注册到容器中的。此时以contextAttribute属性的值为键，在ServletContext中查找上下文，查找得到，说明上下文已经以别的方式初始化并注册在contextAttribute下，直接使用。
  • 检查wac变量的引用是否为null，如果wac==null成立，说明2、3两步中的上下文初始化策略都没成功，此时调用createWebApplicationContext(rootContext)，建立一个全新的以rootContext为父上下文的上下文，作为SpringMVC配置元素的容器上下文。大多数情况下我们所使用的上下文，就是这个新建的上下文。
  • 以上三种初始化上下文的策略，都会回调onRefresh(ApplicationContext context)方法（回调的方式根据不同策略有不同），onRefresh方法在DispatcherServlet类中被覆写，以上面得到的上下文为依托，完成SpringMVC中默认实现类的初始化。
  • 最后，将这个上下文发布到ServletContext中，也就是将上下文以一个和Servlet类在web.xml中注册名字有关的值为键，设置为ServletContext的一个属性。你可以通过改变publishContext的值来决定是否发布到ServletContext中，默认为true。

以上面6点跟踪FrameworkServlet类中的代码，可以比较清晰的了解到整个容器上下文的建立过程，也就能够领会到FrameworkServlet类的设计目的，它是用来建立一个和Servlet关联的Spring容器上下文，并将其注册到ServletContext中的。跳脱开SpringMVC体系，我们也能通过继承FrameworkServlet类，得到与Spring容器整合的好处，FrameworkServlet和HttpServletBean一样，是一个可以独立使用的类。整个SpringMVC设计中，处处体现开闭原则，这里显然也是其中一点。

Tomcat 概述

发表于 2017-09-08

Tomcat 实现了Servlet规范、JSP规范，支持对这两者的解析，还支持JSTL表达式语言、和Java WebSocket。相对于apache/nginx服务器(web server)，Tomcat属于Web容器(Web Container). 类似的Web容器还有Jetty, Resin等。

容器是参与管理内部事物整个生命周期的，比如Tomcat负责整个Servlet的生命周期(LifeCycle),包含什么时候创建Servlet,什么时候初始化内部的一些资源，请求处理的时候执行什么方法，什么时候销毁等。类似的，Spring的Ioc容器管理整个Spring Bean的生命周期。而像Apache之类的Web服务器只能处理HTTP协议，例如我们在请求HTML,图片等这些静态资源的时候，可以通过Web服务器来完成。

两者的区别就是Tomcat不仅可以用于处理动态资源，处理Servlet、JSP，也能处理静态资源，而Apache只能处理静态资源。

Tomcat目录结构

bin: 存放一些启动停止的脚本文件
conf: 配置文件
lib: Tomcat依赖的一些类库
logs: 存放运行过程中生成的日志文件
webapps: 存放需要Tomcat运行的web应用
work: 存放运行过程中jsp编译生成的java文件，和class文件

我们自己创建的web应用都会有个web.xml配置文件，而在tomcat的conf目录下也有个web.xml文件，其作用是此Tomcat内的所有Web应用的全局配置文件。

web.xml中有一些默认配置。处理静态资源的DefautServlet，处理JSP文件的JspServlet，session默认超时时间30分钟，请求一个应用的根目录时，会默认显示首页（index.jsp, index.html,index.htm)。当然，web应用自身的web.xml优先级要更高。

Go channel

发表于 2017-09-06

channel 用于多个goroutine间发送和接受共享的数据，达到数据共享的目的。

ch:=make(chan int) //默认大小0，无缓冲channel
ch:=make(chan int, 2) // 有缓冲channel
close(ch) //关闭channel后不能往channel发送数据，可以接收数据
cap(ch) //channel的最大容量
len(ch) // channel现有元素个数

无缓冲channel不能保存任何值。有缓冲channel是一个容量为n的队列。当队列满的时候，向channel发送数据会阻塞，当队列空的时候，接收数据会阻塞，

实战

使用两个goroutine交替打印序列，一个goroutinue打印数字， 另外一个goroutine打印字母， 最终效果如下 12AB34CD56EF78GH910IJ。

问题很简单，使用 channel 来控制打印的进度。使用两个 channel ，来分别控制数字和字母的打印序列， 数字打印完成后通过 channel 通知字母打印, 字母打印完成后通知数字打印，然后周而复始的工作。

Golang 基础

发表于 2017-09-03

定义变量

var name string
name := nenghaizi

以上两种基本方式，其中第二种只能用在函数内部，一般用var定义全局变量。

定义常量

const Pi = 3.1415926

基本类型

数值类型

整数类型

  • int8, int16, int32, int64
  • uint8, uint16, uint32, uint64

浮点类型

  • float32
  • float64 默认

字符

string可以进行切片操作

布尔类型

var isActive bool  //默认false

上述类型为原始类型，不可改变，函数调用时传递的是这些值的副本

引用类型

包括slice，map，interface，chan，函数类型

引用类型的变量保存的是一个指针，指向底层数据结构，函数调用时传递的是这个指针的副本，而底层数据结构并没有传递。

本质上，我们可以理解函数的传递都是值传递，只不过引用类型传递的是一个指向底层数据的指针，所以我们在操作的时候，可以修改共享的底层数据的值，进而影响到所有引用到这个共享底层数据的变量。

slice

既然是引用类型，slice的指向是底层array。相当于包含3个字段的数据结构，一个指向数组的指针，一个slice长度，一个slice容量。

声明

aslice := make([]int, 4, 8)  //长度4，容量8指的是底层array的长度
bslice := []int{1,2,3,4} //没有像声明array一样指定长度，slice推断出长度为4，且容量也为4

比较：
var nilSlice []int // 指向array的指针为nil，还未初始化
cslice := []int{} // 指针肯定不为nil，代表一个空集合

slice 通过append添加内容，动态扩充底层array的容量。

从同一个array引出的slice，其中一个slice改变了某个值，则其他slice以及array中的改值都会改变。

函数调用时传递的是slice副本，即3个字段的数据结构。对其修改后，所指向的底层数组也会随之改变。

Map

dict := make(map[string]int) // 声明一个map，可以添加key:value使用了。
dict := map[string]int{} //一个空map

var dict map[string]int // nil, 并没有指向任何内存地址，需要开辟一块内存才能使用，如下。
dict = make(map[string]int)  或 dict = map[string]int{}

函数调用传递的同样是map的引用副本。

make 与 new

make只能用于内建类型（map、slice 和channel）的内存分配。new用于各种类型的内存分配。

new(T)分配了零值填充的T类型的内存空间，并且返回其地址，即一个*T类型的值。这个指针指向的内容的值为零。

make返回一个有初始值(非零)的T类型，而不是*T。

结构类型

函数调用过程，结构体传递的是其本身以及里面的值的副本。

通过传递结构体的指针, 可以达到我们的预期修改age的目的。

但是函数传递结构体时，如果结构体中含有引用类型，如Map，是会产生修改的。

函数方法

定义一个方法需要指定一个接收者。接收者分：指针接收者和值接收者。

使用值类型接收者定义的方法，在调用的时候，使用的其实是值接收者的一个副本，所以对该值的任何操作，不会影响原来的类型变量。

指针接收者传递的是一个指向原值指针的副本，指针的副本，指向的还是原来类型的值，所以修改时，同时也会影响原来类型变量的值。

调用指针接收者方法时，使用值或者指针都可以，Go的编译器自动会帮我们取指针。同样调用值接受者方法时，也可以使用指针，Go编译器自动会解引用。

在调用方法的时候，传递的接收者本质上都是副本，只不过一个是这个值副本，一是指向这个值指针的副本。指针具有指向原有值的特性，所以修改了指针指向的值，也就修改了原有的值。我们可以简单的理解为值接收者使用的是值的副本来调用方法，而指针接收者使用实际的值来调用方法。

接口

实现一个接口的方法，可以使用指针接收者和值接收者。二者区别为：实体类型以值接收者实现接口的时候，不管是实体类型的值，还是实体类型值的指针，都实现了该接口。实体类型以指针接收者实现接口的时候，只有指向这个类型的指针才被认为实现了该接口。

govendor Golang依赖管理

发表于 2017-09-03

golang 项目默认从$GOPATH中查找依赖，不同用户不同时间从外部获取的依赖版本可能不同。

govendor工具将项目依赖的外部包放到项目下的 vendor 目录下，通过 vendor.json 文件来记录依赖包的版本。

govendor 安装

go get -u -v github.com/kardianos/govendor

golang包类型如下

状态 缩写 含义
+local l 本地包，即项目自身的包组织
+external e 外部包，在$GOPATH中，但不在vendor目录下
+vendor v 被govendor管理，即在vendor目录下
+std s 标准库中的包
+unused u 未使用的包，即包在vendor目录下，但项目并没有用到
+missing m 代码引用了依赖包，但该包并没有找到
+program p 主程序包，意味着可以编译为执行文件
+outside 外部包和缺失的包
+all 所有的包

使用

项目中引用了外部包，在项目根目录下执行

govendor init

来初始化vendor 目录，会生成一个vendor目录，目录中有一个vendor.json文件，用来存放项目依赖信息。执行

govendor add +external

将项目所有依赖信息添加至vendor.json文件中。

govendor add PKG_PATH  //添加指定包路径信息

其他命令：

  • govendor list 列出所有依赖
  • govendor update 从$GOPATH更新依赖包到vendor目录
  • govendor remove 从 vendor 管理中删除依赖
  • govendor sync 本地存在 vendor.json 时候拉去依赖包，匹配所记录的版本
  • govendor get 拉取依赖包到 vendor 目录

docker 常用命令

发表于 2016-10-22

镜像

  • docker pull [选项] [Docker Registry地址]<仓库名>:<标签>
  • docker pull ubuntu:14.04

上面的命令中没有给出 Docker Registry 地址，因此将会从 Docker Hub 获取镜像。而镜像名称是 ubuntu:14.04，因此将会获取官方镜像 library/ubuntu 仓库中标签为 14.04 的镜像。

  • 在某个镜像的基础上运行容器
  • 命令举例：docker run -it –rm ubuntu:14.04 bash

每次执行 docker run 命令都会创建新的容器，建议一次创建后，使用 docker start/stop 来启动和停用容器。

-i：交互式操作   
-t：终端    
--rm: 容器退出后随之将其删除    
ubuntu:14.04：表示用这个镜像为基础启动容器
bash：命令
exit命令退出容器
  • docker images

列出已经下载下来的镜像。包含几个字段：

REPOSITORY TAG IMAGE ID CREATED SIZE
  • 命令举例：docker run –name webserver -d -p 80:80 nginx

在nginx镜像上运行一个容器，命名为webserver。-d代表后台运行并输出容器ID，-p映射端口80

启动这个容器后，可以通过exec进入容器修改内容

  • docker exec -it webserver bash

进入后可以执行：cat /etc/os-release 查看系统版本。apt-get update apt-get install vim 这两条命令可以安装vim。上述即是对容器做了改动，执行：

  • docker diff webserver

查看都有哪些变化

  • docker commit –author “wxf” –message “修改了默认网页” webserver nginx:v2

运行容器时候做的一些修改会记录到容器存储层，commit可以将存储层保存成新的镜像（nginx:v2 代表 仓库名：标签）
